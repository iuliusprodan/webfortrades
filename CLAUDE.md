# WebForTrades — Claude Code project instructions

WebForTrades is a single-operator pipeline that finds UK trade businesses with weak or no
website, builds each a bespoke one-page Next.js site **on spec**, deploys it to Vercel under a
business-branded alias, and pitches it (WhatsApp manual, email automated). `leads.db` (SQLite)
is the single source of truth for lead state.

This file is **principles + pointers**, auto-loaded every session. Detail lives in skills
(see "Skill pointers"). Sub-directory `CLAUDE.md` files **extend** this one; they never override
an invariant below.

---

## Architectural invariants (non-negotiable)

These are decisions, not open questions. If a task seems to require breaking one, stop and surface
the tension — do not work around it silently.

1. **ARCH-1 — Open Design is RETIRED (2026-06-13).** The bake-off established Path B (Claude-direct from
   a frozen brief) as the production pipeline. Do not use Open Design for new builds. See
   `docs/claude-migration/bake-off-results.md`.
2. **ARCH-2 — Claude writes the understanding.** Strategy, section-plan, and voice are generated by
   Claude from evidence. The heuristic Node generators (`site_strategy.ts`, `section_planner.ts`,
   `voice_discovery.ts`) survive only as fallback/validator. This is the core of the migration.
3. **ARCH-3 — New builds use Claude-direct only.** The legacy template `build.ts` remains solely as a
   reference and rebuild escape hatch for the 27 existing deployed sites; it is not used for new builds.
   Do not delete the template path.
4. **ARCH-4 — Sequential builds for now.** Bake-off and the first 2-3 new-pipeline batches run
   sequentially on the main thread. Port-pool machinery stays in tree but is not invoked on the new
   path. Concurrency is a later decision.
5. **ARCH-5 — WhatsApp is permanently manual.** No OpenWA, no automated WhatsApp send, ever, for any
   reason. The pipeline writes `data/outreach-queue-whatsapp.md`; the operator sends by hand.
6. **ARCH-6 — Email has one chokepoint.** All email sends go through `scripts/outreach/send_email.ts`,
   which reads `sending_enabled` fresh from `config.yaml` at call time and throws if false. No
   override flags, no waive paths, no other SMTP caller.
7. **ARCH-7 — `config.yaml` is read-only at runtime.** Code reads it; code never writes it. Runtime
   config mutation is what defeated the outreach kill-switch (see `outreach-gating-rootcause.md`).
   *Enforced by two layers:* a runtime startup assertion in the pipeline entry script that fails if
   `config.yaml`'s mtime changes during the run, and a static check in the check pipeline that fails
   if any `scripts/` file calls `fs.writeFile*`/`appendFile` against `config.yaml`.
   *(Proposal, decide later: also `chmod 0444 config.yaml` so the OS denies writes outright — strongest
   guard, mild cost that manual edits need `chmod +w` first.)*
8. **ARCH-8 — Pitches are Claude-written and evidence-anchored.** WhatsApp queue messages are
   generated per lead from `pitch-insight.json` + `site-strategy.json` + `source-evidence.json`,
   each anchored on one verifiable detail. A batch swap-test runs before the queue file is written.
   No generic templates (they produced the 14-send slop).
9. **ARCH-9 — Manual outreach state is local-only.** `outreach:mark-sent` / `outreach:mark-reply`
   update `leads.db` and the logs with zero network calls.
10. **ARCH-10 — Skills are project-local by default.** WebForTrades-specific knowledge lives in
    `.claude/skills/` in this repo. `~/.claude/skills/` holds only cross-project skills.
11. **ARCH-11 — No AI image generation.** Gemini/Nano-Banana is retired and out of scope. Revisit
    only if a real hero-only failure later has no other fix.
12. **ARCH-12 — `qualification.*` gates the queue, not a send.** The WhatsApp disqualifiers
    (non-CONTACTABLE, landline-only, no contact method) now decide queue *inclusion*, since there is
    no automated send to gate.

## Banned phrases & patterns (scannable reference)

Source of truth: `scripts/copy_voice_constants.ts` + the site-design skill. Enforced by the check
pipeline (`run_site_source_checks.ts`). This is the quick reference; the skill holds the rationale.
If this list and `scripts/copy_voice_constants.ts` ever disagree, the constants file wins. Flag the
drift as a bug — do not silently sync either direction.

**Punctuation**
- No em dash `—`, en dash `–`, horizontal bar `―`, or ASCII ` -- `. Use ` - `, comma, or full stop.

**Hero (headline must be trade-led: trade + location + service/outcome)**
- Banned: customer review/testimonial quote as headline; first-person narrative ("I recently…");
  owner name or personal backstory; generic platitude; long brand mission statement.
- Banned legacy headings: "Plumbing sorted properly", "Questions before you ring.",
  "One van. One trade. A name on a list.", "{N} services. Done plainly.", "A note from {name}",
  "Recent work in {area}.", "Pick up the phone, or write.", "Need this sorted? Get a free quote."

**Reviews**
- Exactly one review section per page; no duplicate "What customers say" + review wall.
- No reviewer first name as a section headline (the "Steve" anti-pattern).
- Attribution only as "Google review" / "Customer on Google".
- Every named person in a review must match `brief.owner_name`/`team_member_names`, else swap it.

**Services (3-6 real trade services, each with an evidence-based description)**
- Never Google Places categories, CTAs, or coverage areas as service items.
- No inline `<svg>`, `lucide-react`, or `@heroicons/*` in service/process UI (use numbers, dots,
  two-letter marks, or nothing).

**Meta-provenance (never explain where the copy came from)**
- Banned substrings incl.: "described in google reviews", "drawn from the verified google listing",
  "from google places", "the verified facebook page", "google reviews name".
- Banned sentence starts: "reviews describe/mention/note…", "customers mention…", "sourced from google".
- Banned defensive filler: "not stock shots", "verified customer proof", "evidence only",
  "trust signals", "third-party verification", "this preview form does not submit live".

**Stats**
- Banned sub-labels: "sourced from google", "direct line", "direct contact line".
- Only sourced stats (Google rating, review count, years, certifications). Never photo counts.

**Sticky CTA**
- Quote-only. Allowed labels: "Get a quote", "Get quote", "Request a quote", "Get a free quote".
- Never a phone number or "Call" in the sticky CTA.

**Generic / salesy (avoid everywhere, copy and outreach)**
- "your trusted partner", "one-stop shop", "go the extra mile", "second to none",
  "no job too big or small", "competitive prices", "professional service", "satisfaction guaranteed",
  "quality you can rely on", "explained plainly", "we pride ourselves", "tailored solutions",
  "your local … experts".
- Outreach voice also bans: amazing, premier, professional, trusted, leading, top-rated, expert;
  no exclamation marks; no emoji.

**Badges (claim only if backed by `brief.certifications` or a verified directory probe)**
- Gas Safe, NICEIC, NAPIT, Part P, "fully insured", Checkatrade Vetted/member, TrustATrader
  (Approved), Which? Trusted Trader.

## Verification discipline

Never report success from an input, a log, or a cached read. Verify the actual output. Three real
failures from this codebase and the rule that would have caught each:

1. **Deploy "shipped" on a cached fetch.** A WebFetch returned a stale/cached page and that was taken
   as proof the new build was live.
   → *Rule:* confirm a deploy by fetching the canonical alias with cache busting and matching the
   **exact build marker just deployed** (plus business name + phone, and live image URLs return 200).
2. **Logs trusted while the logger was deceptive.** The outreach log recorded `sending_enabled=false`
   on 14 real sends because the sender flipped the flag and logged the post-reset value.
   → *Rule:* never treat a self-reported field as truth for an irreversible action; cross-check against
   an independent signal (gateway message ID, `config.yaml` mtime, DB row), and re-read the live file
   rather than trusting a remembered flag.
3. **"9 distinct accents PASS" on identical pages.** Palette tokens differed per site but every rendered
   surface stayed the default template — the check compared the input token, not the output.
   → *Rule:* verify the rendered DOM/computed style, not the config that fed it, and apply the swap
   test (could this page belong to another business? then it failed).

General: if you cannot show the proof (a `curl`/`grep`/screenshot/DB query), you have not verified it —
say "unverified," do not say "done."

## Skill pointers

These land after the skill port (`03-skill-port-plan.md`); names may shift during that step.

**Project-local (`.claude/skills/`)**
- `webfortrades-site-design` — read before creating/rebuilding/reviewing any site (evidence → strategy
  → section plan → design → copy → review gate → pitch; the swap test; banned-phrase rationale).
- `webfortrades-enrich` — read when gathering a lead (Google Places, Facebook media policy, directory
  probes, identity/homonym verification, source-evidence schema).
- `webfortrades-deploy-verify` — read before/after any deploy (alias canonicalization, live build-marker
  + image-200 checks, state-sync between `deploy.json` and `leads.db`).
- `webfortrades-outreach` — read when generating the WhatsApp queue or sending email (queue spec, pitch
  anchoring, email chokepoint, `mark-sent`/`mark-reply`, no automated WhatsApp).

**User-global (`~/.claude/skills/`)**
- `verification-discipline` — cross-project: prove success against live output, never inputs or logs.
- `evidence-before-edits` — cross-project: read the target and its provenance before changing/deleting.
